lua_of_ocaml project review
===========================

Scope
-----
This is a whole-project review pass focused on bad-looking code, logic
risks, and structural problems. No source files were changed during the
review.

Working tree state observed
---------------------------
The repository already had uncommitted changes in:

- runtime/lua/misc.lua
- test/ffi/ffi_test.lua
- test/ffi/roundtrip.lua
- test/ffi/roundtrip.ml
- test/ffi/lua_stubs.c

Those were treated as current project state.

Build and test status
---------------------
Plain `make test` fails on this machine before running the tests because
the shell `ocamlc` is 5.4.1 while the active opam switch is 5.3.0+BER, and
the installed js_of_ocaml compiler interface belongs to the opam compiler.

Using the intended switch works for the build:

  opam exec -- dune build

The full test script then reaches runtime execution but fails:

  opam exec -- bash test/run_all.sh

It fails at `smoke/hello (run)`. Running the generated Lua directly prints
`hello` and then fails with:

  attempt to call global 'caml_atomic_load' (a nil value)

Findings
--------

1. Runtime aliases are assigned before their functions exist

File: runtime/lua/misc.lua

`caml_atomic_load = caml_atomic_load_field` and
`caml_atomic_cas = caml_atomic_cas_field` are assigned before
`caml_atomic_load_field` and `caml_atomic_cas_field` are defined. In Lua
this stores nil, not a late binding. Generated programs that call
`caml_atomic_load` fail at runtime.

This is the immediate cause of the full test failure.

Suggested fix:
Move the alias assignments below the function definitions, or define wrapper
functions instead:

  function caml_atomic_load(...) return caml_atomic_load_field(...) end
  function caml_atomic_cas(...) return caml_atomic_cas_field(...) end

2. Generated compiler variables are global, not local

File: compiler/lib/generate_lua.ml

The comment says "Collect all vars from all blocks, declare as local at top",
but the implementation emits assignments like:

  x = nil

instead of:

  local x = nil

In Lua this creates or mutates globals. That is structurally risky for
closures, repeated `_main()` calls, and name collisions between generated
program state and runtime/global functions.

Suggested fix:
Emit `L.Local` declarations for generated variables. This may require
chunking declarations if the variable count gets too large for Lua 5.1
limits.

3. Array/vector indexing is inconsistent between generator and runtime

Files:
- compiler/lib/generate_lua.ml
- runtime/lua/array.lua

Inline generated operations do not match runtime helpers:

- `Vectlength` emits `#v - 1`
- `caml_vect_length` returns `(#v - 1) * 2`
- inline `Array_get` indexes with `i + 1`
- `caml_array_get` indexes with `floor(i / 2) + 2`

OCaml ints are tagged in this runtime, so the inline versions look wrong.
This can produce incorrect behavior even if syntax tests pass.

Suggested fix:
Either make the generator call the runtime helpers consistently, or update
the inline lowering to use the same tagged-index conventions.

4. Exception support is mostly stubbed

Files:
- runtime/lua/fail.lua
- compiler/lib/generate_lua.ml

`caml_raise` currently returns `0` instead of raising. The generator emits
`caml_raise(exn)` as a bare expression statement, so many raises will be
silently swallowed instead of transferring control.

The Pushtrap/Poptrap path is also incomplete and does not look like a
faithful exception model yet.

Suggested fix:
Use Lua `error`/`pcall` consistently for exceptions, preserve OCaml exception
values through the thrown payload, and add behavioral tests that assert both
handled and unhandled exceptions.

5. Several core value translations are placeholders

File: compiler/lib/generate_lua.ml

The following translations are clearly incomplete:

- non-byte NativeString becomes an empty string
- Float becomes `{253, 0}`
- Float_array becomes `{253, 0}`
- Int64 becomes `0`

These are acceptable as temporary scaffolding, but they should be marked and
tested as unsupported. Right now they can make programs compile and run with
wrong values.

Suggested fix:
Fail loudly for unsupported constants, or implement the runtime
representation before claiming support.

6. Control-flow structuring has unused/incomplete pieces

File: compiler/lib/generate_lua.ml

`edge_kind` defines `Loop`, `Exit_loop`, and `Forward`, but only `Loop` is
meaningfully used. `compile_conditional` receives `_scope_stack` and then
branches with an empty stack, which discards loop/scope context.

This makes loop exits, forward edges, and merge handling suspicious. It may
work for very simple programs while failing on more complex bytecode CFGs.

Suggested fix:
Add CFG-focused tests with loops, nested loops, early exits, and branches
that merge. Then either complete the structural algorithm or simplify it to
a more explicit dispatcher/state-machine approach.

7. Tests mostly check syntax, not correctness

Files:
- test/run_all.sh
- test/compiler/*.ml

Most compiler tests only assert that generated Lua parses with `loadfile`.
The OCaml test programs usually `ignore` computed results, so incorrect
generated logic can still pass.

Runtime smoke tests only check process exit status. They do not compare
generated Lua output with native OCaml output.

Suggested fix:
For each compiler test, run both the bytecode program and generated Lua, then
compare stdout/stderr/exit status. Replace `ignore`-only programs with tests
that print or assert expected values.

8. Documentation and Makefiles contain stale script names

Files:
- README.md
- test/compiler/Makefile

The README says to run:

  bash test/run_tests.sh

but the repo has:

  test/run_all.sh

`test/compiler/Makefile` also references `../run_tests.sh`, which does not
exist.

Suggested fix:
Update documentation and Makefiles to use `run_all.sh`, or add a compatibility
wrapper script.

Priority order
--------------

1. Fix `caml_atomic_load` and `caml_atomic_cas` alias ordering in
   runtime/lua/misc.lua.
2. Make `make test` run in the correct opam environment, or document that
   tests must be run via `opam exec -- make test`.
3. Strengthen the test suite to compare actual behavior, not just Lua syntax.
4. Fix local variable declaration in generated Lua.
5. Align inline array/vector lowering with runtime helpers.
6. Implement or explicitly reject unsupported constants such as Float and
   Int64.
7. Rework exception semantics and add focused exception tests.
8. Audit control-flow generation with complex CFG tests.

Overall assessment
------------------
The project structure is understandable and small, but the compiler/runtime
contract is not yet stable. The biggest structural issue is that tests allow
large logic gaps to pass. The immediate runtime failure is easy to fix, but
the next useful step should be making tests behavioral so future compiler
changes expose real semantic regressions.
